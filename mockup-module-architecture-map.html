Architecture Reference · YouTube CTR Pipeline

각 모듈은 무슨 일을 하고
어디를 담당하는가

Autoresearch의 최종 목표는 "이 사용자에게 이 영상을 보여주면 클릭할까?"를 예측하는 CTR 모델입니다. 그런데 실제 사용자 클릭 로그가 없으므로, 가상의 사용자를 만들고 · 실제 트렌딩 영상을 수집한 뒤 · 둘을 만나게 해 클릭 로그를 합성하고 · 그 로그로 모델을 학습합니다. 아래는 그 네 단계를 이루는 모듈 각각이 무엇을 맡는지 정리한 지도입니다.

가상 유저 생성 ＋ 트렌딩 영상 수집 → 클릭 로그 합성 → CTR 모델 학습
범위 main · autoresearch/ + src/ 런타임 Python 3.12 · Airflow · Cloud Run 공통 구조 schema · pipeline · generator · Protocol

파이프라인 스테이지 (색상)

  • youtube_collection — 트렌딩 영상을 모아 레이크에 쌓는다
  • virtual_users — 학습에 쓸 가상 사용자를 만든다
  • action_logs — 유저×영상으로 클릭 로그를 합성한다
  • jobs · dags — 위 작업을 정해진 시각/방식으로 실행한다
  • src · feature_repo — 로그로 CTR 모델을 학습한다 (별도 스택)

수정 시 영향 (blast radius)

  • 높음 — 스키마/데이터 계약. 다수 모듈 + 영속 parquet + 하류까지 파급
  • 중간 — 파이프라인 진입점. DAG/CLI 계약, 여러 호출부 영향
  • 낮음 — 내부 헬퍼/독립 유틸. 국소적, 호출부 제한적

02

youtube_collection

수집 파이프라인 담당 영역 — 모델이 "보여줄 영상 후보"로 쓸 실제 KR 트렌딩 영상을 매일 확보하는 일. YouTube에서 데이터를 안전하게 긁어와, 정규 스키마로 다듬고, 데이터 레이크에 날짜별로 쌓습니다. 여기가 파이프라인의 영상 쪽 원천입니다.

schema.py

영향 높음

"트렌딩 영상 한 편"의 표준 정의서. 영상에 어떤 컬럼이 있고(제목·조회수·채널·태그 등 29개) 각 값이 어떤 타입·범위여야 하는지를 한 곳에 못박아, 뒤따르는 모든 모듈이 같은 모양을 공유하게 합니다.

담당필드·타입 검증, 스키마 버전 식별, KR 스코프 상수 제공

비담당값을 실제로 파싱·변환(transform), 파일로 적재(load)

IN
없음 (계약 정의 · 최하위 계층)
OUT
TrendingVideo pydantic 모델
계약
SCHEMA_VERSION="youtube_trending_kr_v1" · TARGET_COUNTRY="KR" · 통계 필드 ge=0 · 같은 영상 매일 별도 행(dedup 금지 — 일자별 델타가 피처)
수정 시 영향필드/타입 변경 시 transform · load(_SCHEMA) · backfill 전부 + 영속 parquet + 하류 action_logs.video_source까지. 버전 올려야 함.

transform.py

영향 중간

서로 다른 원천을 하나의 표준으로 맞추는 어댑터. 과거 Kaggle CSV와 실시간 API 응답은 형식이 제각각인데, 이 모듈이 둘 다 같은 통로로 통과시켜 날짜·타입·국가를 정규화한 TrendingVideo로 바꿉니다.

담당타입 강제 변환, 날짜 포맷 정규화, naive→UTC, KR-only 강제

비담당API 호출·페이지네이션(fetch), 파일 IO(load)

IN
raw dict (API item / Kaggle row) + collected_at
OUT
TrendingVideo
계약
normalize_api_item(video_item, channel_item, category_map, *, collected_at, region_code="KR") · normalize_kaggle_row(row, collected_at) · KR 외 ValueError
수정 시 영향변환 규칙(날짜/bool/tags)이 바뀌면 fetch · backfill 출력과 적재 데이터 품질에 직접 반영.

fetch.py

영향 중간

영상을 실제로 긁어와 완성 레코드로 조립하는 수집 지휘자. 트렌딩 목록을 페이지 단위로 받아오고, 각 영상의 채널 정보와 카테고리 이름을 추가로 조회해 하나로 합칩니다. DAG이 매일 호출하는 진입점.

담당페이지네이션, 채널 50개 배치 조회, 카테고리 id→이름, dedup

비담당HTTP·재시도·Key 회전(client), 정규화(transform), 적재(load)

IN
3개 callable list_videos/channels/categories (client 주입)
OUT
list[TrendingVideo]
계약
collect_trending(list_videos, list_channels, list_categories, *, collected_at, region_code="KR", max_results=200) · googleapiclient 직접 import 안 함(주입 seam)
수정 시 영향시그니처 변경 시 dags/youtube_trending_kr_daily 호출부 수정. 쿼터(~7 units/일) 로직 영향.

client.py

영향 중간

API 장애·IP밴에도 수집이 죽지 않게 하는 방탄 계층. fetch에게는 "그냥 부르면 되는 함수 3개"로 보이지만, 내부에서는 재시도·API Key 교체·IP밴 감지·차단기(circuit breaker)를 겹겹이 돌려 실패를 흡수합니다.

담당지수 백오프 재시도, Key 회전, IP밴 판정, breaker, 프록시 경로 전환

비담당데이터 조립·정규화(fetch/transform), 프록시 서버 자체(proxy/)

IN
keys: list[str], proxy_url
OUT
YouTubeCallables (NamedTuple)
계약
ResilientYouTubeClient(...).make_callables() · Verdict enum · CollectionExhausted · 프록시 규약 /youtube/v3/{resource} + X-Goog-Api-Key
수정 시 영향fetch와 callable 계약, proxy/app.py와 HTTP 경로·헤더 계약 공유. 어느 쪽 변경도 짝 맞춰야 함.

load.py

영향 높음

완성된 하루치 영상을 레이크에 떨어뜨리는 적재기. 고정된 컬럼 스키마의 parquet 한 개를 dt=날짜 폴더에 씁니다. 같은 날 다시 돌려도 덮어쓰기라 안전(멱등)하고, 로컬/GCS를 같은 코드로 처리합니다.

담당고정 스키마 parquet 쓰기, 로컬/GCS 분기, 멱등 덮어쓰기

비담당정규화(transform), 스케줄링·버킷 결정(DAG)

IN
list[TrendingVideo], base_path, partition_date
OUT
{base_path}/dt=YYYY-MM-DD/part-0.parquet 경로
계약
write_partition(videos, base_path, partition_date, *, filesystem=None) · _SCHEMA(29컬럼, ts=us/UTC) · None=로컬 / GcsFileSystem=GCS
수정 시 영향경로 규칙/스키마 변경은 backfill, DAG, 이 레이크를 읽는 action_logs.video_source 전부에 파급.

backfill.py

영향 낮음

과거 데이터를 한 번에 채워 넣는 소급 적재기. 실시간 수집(fetch/client) 없이, Kaggle 글로벌 덤프에서 KR 행만 골라 트렌딩 날짜별 파티션으로 레이크를 소급 구성합니다. 초기 세팅용 1회성 경로.

담당KR pushdown 필터 읽기, per-row 예외 격리, 날짜별 그룹핑

비담당실시간 API 수집(과거 데이터 전용)

IN
source_path (Kaggle parquet), base_path
OUT
쓰인 총 행 수 int (dt별 파티션)
계약
backfill_from_parquet(source_path, base_path, *, collected_at=None, filesystem=None)
수정 시 영향load · transform.COUNTRY_ALIASES에 의존하나 이 모듈을 읽는 하류는 없음.

proxy/app.py

영향 중간

IP밴을 피하기 위한 우회 출구(egress seam). 수집 서버의 IP가 차단되지 않도록, YouTube 요청을 Cloud Run을 거쳐 내보내는 얇은 중계기입니다. 판단은 안 하고 그대로 전달만 하는 "dumb forwarder".

담당경로 화이트리스트 전달, path traversal 차단, key query 차단, liveness

비담당Key 상태·회전(client가 담당), 재시도·에러 분류

IN
HTTP GET + X-Goog-Api-Key 헤더
OUT
upstream JSON passthrough / 400·502·503
계약
GET /youtube/v3/{rest_path} · GET /health · key query 차단(마스킹 불변량) · UNHEALTHY_THRESHOLD=3
수정 시 영향코드 의존 없음(별도 배포). 유일한 결합은 client.py와의 HTTP 경로·헤더 계약. 규약 변경 시 양쪽 동시 수정.
03

virtual_users

생성 파이프라인 담당 영역 — 실제 사용자 데이터가 없으니 학습에 쓸 가상의 시청자를 만들어내는 일. 실존 인물 통계(NVIDIA 페르소나)를 표본으로 뽑아, LLM으로 "이 사람은 유튜브를 어떻게 볼까"를 상상해 붙입니다. 파이프라인의 사용자 쪽 원천입니다.

schema.py

영향 높음

"가상 유저 한 명"의 표준 정의서. 나이·성별·직업·관심 카테고리·시청 시간대 등 한 유저가 가질 속성과 생성 요청 파라미터를 정의합니다.

담당유저·요청·프로필 모델 정의, 버전 상수, 연령 버킷 규칙

비담당LLM 값 채우기(glm_generator), 저장(pipeline)

IN
없음 (계약 정의)
OUT
VirtualUser · GenerationRequest · YouTubeProfile 등
계약
GENERATION_SCHEMA_VERSION="virtual_user_schema_v1" · age_bucket_for_age(age) · sex ∈ {male,female}
수정 시 영향필드 변경 시 glm_generator · pipeline(33필드 parquet) + 하류 action_logs가 읽는 user_id 계약까지.

영향 낮음

04

action_logs

클릭 로그 합성 담당 영역 — 파이프라인의 심장부. 앞의 두 원천(가상 유저 · 트렌딩 영상)을 만나게 해, "이 유저가 이 영상들을 보면 뭘 클릭할까"를 LLM으로 판단하고 impression/click/view/like 이벤트 로그를 합성합니다. 이 로그가 곧 CTR 모델의 학습 데이터(라벨 원천)입니다.

schema.py

영향 높음

"이벤트 한 줄"의 표준 정의서. 누가(user) · 언제 · 어떤 영상에(video) · 무슨 행동(impression/click/view/like)을 했는지를 정의합니다. 목표 CTR·후보 수 같은 생성 규칙도 여기 상수로 고정.

담당이벤트·요청·shard manifest 모델 정의, CTR/후보 상수

비담당clicked 라벨 저장(하류가 impression⋈click로 파생)

IN
없음 (계약 정의)
OUT
EventLog · EventGenerationRequest · ActionLogShardManifest
계약
ACTION_LOG_SCHEMA_VERSION="action_log_schema_v1" · PROMPT_VERSION="action_log_ctr_v4" · target_ctr=0.02 · candidates_per_user=24
수정 시 영향이벤트 필드 변경은 pipeline(EVENT_LOG_PARQUET_SCHEMA) · daily + 영속 로그 + CTR 학습셋 빌더까지.
